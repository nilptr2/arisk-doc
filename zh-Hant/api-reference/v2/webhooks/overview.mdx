Arisk V2 透過 Webhook 向您配置的伺服器傳送實時 HTTP POST 請求，通知監控地址的風險等級變動或風險交易。

## 安全性驗證

為確保訊息來源的真實性和完整性，Arisk 傳送的每個 Webhook 請求都包含數字簽名。我們強烈建議您在處理請求前進行驗證。

### HTTP 請求頭

每個 Webhook 請求都會包含以下關鍵 Header：

| Header | 說明 | 示例 |
| :--- | :--- | :--- |
| **X-Arisk-Signature** | RSA-PSS 數字簽名（Base64 編碼） | `aGVsbG8gd29ybGQ=...` |
| **Authorization** | Bearer Token（僅在配置了 `auth_key` 時存在） | `Bearer your-secret-token` |

    *   **簽名內容**: 請求 Body 的原始 JSON 字串
    *   **公鑰**: 使用下方的 Arisk RSA 公鑰驗證 `X-Arisk-Signature` 中的簽名。

## 最佳實踐

- **快速響應（必選）**
  您的伺服器在接收到 Webhook 請求後，應立即返回 HTTP 200 OK 狀態碼。
  - **返回資料**：響應體（Body）可以是空內容，也可以是一個簡單的 JSON（例如 `{"message": "received"}`）。我們的服務僅校驗 HTTP 狀態碼，不會解析響應體內容。
  - **超時風險**：請儘量避免在同步請求中執行耗時操作，否則可能觸發傳送端的超時重試機制（預設超時通常較短）。

- **非同步處理（推薦）**
  建議採用“先確認，後處理”的策略。即在校驗簽名透過並接收資料後，立即返回 HTTP 200 響應，隨後將耗時的業務邏輯（如寫庫、級聯通知等）放入後臺佇列非同步處理，以確保介面的高可用性。

- **冪等性保障（關鍵）**
  受網路波動或重試機制影響，您可能會收到重複的 Webhook 通知。請務必在業務層利用 Payload 中的 `alert.id` 欄位作為唯一鍵進行冪等檢查。
